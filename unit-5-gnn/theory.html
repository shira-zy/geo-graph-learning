Beat 3 opener — the GRU explainer (review-pass insert), placed BEFORE the T-GCN unroll so the "GCN inside a GRU" story lands. The no-PyTorch subgroup needs this concretely. Say it plainly: a GRU is a tiny recurrent cell that keeps a running MEMORY vector h_t. At each time step it receives two things — the current step's input x_t (in T-GCN that input is the GCN's spatial output for all nodes at that step) and the previous step's memory h_{t-1} — and it uses two GATES (the update gate z and the reset gate r) to decide how much of the new information to write in versus how much of the old memory to carry forward. Its output is the new memory h_t, which is handed to the next step; that recurrence (the curved arrow) is what lets it integrate a whole window of steps. Per-step I/O in one line: (x_t, h_{t-1}) → h_t. Reassure them: you don't implement the gates, you name the cell and let PyTorch build it. Anticipated question: "GRU vs LSTM?" — same idea (a gated memory); LSTM has an extra cell state and one more gate; GRU is lighter and is what T-GCN uses. This directly sets up the next slide: the GCN handles space WITHIN a step, the GRU (this cell) carries the memory ACROSS steps. Demo tie: Section 4's T-GCN wraps a GCN inside exactly this GRU.

Beat 4 close, first half of the split — Face (c)'s deepest point (digest §3b + synthesis). Lead with the STRONGEST honesty point (dogfood fix): in PyG-Temporal, A3T-GCN's attention is a SINGLE global learned weight vector, softmax'd — the SAME weighting applied to every window, node, and forecast.
